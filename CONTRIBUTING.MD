# CONTRIBUTING.md

## 目的

这份文档不是通用开源模板，而是 `Files Rusted` 的**工程决策记录与协作约定**。

用途：

- 记录关键架构决策
- 记录为什么这样做，而不是只记录做了什么
- 让后续接手者能快速理解当前方向
- 避免同一类问题被反复重新讨论

从现在开始，**每一次重要的架构取舍、状态边界调整、线程模型变化、UI 数据流变化、配置持久化策略变化，都应追加到本文档**。

---

## 当前项目目标

`Files Rusted` 是一个基于 **Rust + Slint** 的资源管理器 / 文件浏览器实验项目。

当前总目标不是“最快堆功能”，而是：

1. 先建立稳定、可扩展、不会拖垮 UI 的底层架构
2. 再逐步补齐交互行为与桌面体验
3. 在大量文件、频繁切目录、复杂交互下仍保持稳定

---

## 当前协作约定

### 1. 先记录决策，再做大改

对于非微小修改，在进入实现前，应先在这里记录：

- 要解决什么问题
- 选了什么方案
- 为什么不选其他方案
- 这个方案的代价是什么

### 2. 优先记录“架构性变化”

重点记录以下类型：

- 线程模型
- 事件循环交互方式
- Slint 模型绑定方式
- 文件系统读取策略
- 导航与选择状态模型
- 配置持久化策略
- 无边框窗口与平台行为处理方式

### 3. 不把临时实现细节写成长期规则

这里记录的是**稳定决策**，不是临时草稿。

---

## 架构底线

以下底线目前视为项目级约束：

1. **绝不阻塞 Slint 事件循环**
   - 磁盘 I/O、大目录遍历、重量级元数据收集必须放到后台线程
   - UI 更新必须通过主线程安全回传

2. **UI 列表必须数据驱动**
   - 文件列表、导航项、面包屑、排序项等都应使用 `ModelRc` / `VecModel`
   - 禁止重新回到硬编码列表结构

3. **先保证一致性，再追求缓存优化**
   - 在目录切换、异步刷新、排序切换中，UI 一致性优先于局部性能投机

4. **界面可以轻，但状态模型必须清楚**
   - 视觉层保持简洁
   - 状态边界不能模糊

### ADR-001：总体重构拆成四个阶段推进

**日期**：2026-03-26
**状态**：已采纳

**决策**：
把当前 13 条大需求拆成四个彼此相关但可分阶段落地的子系统，而不是一次性同时重构：

1. 文件系统与后台加载
2. 交互行为层（双击、右键、路径输入、排序/视图切换）
3. 导航与配置层（收藏夹、设置、持久化）
4. 窗口系统层（无边框、拖拽移动、边缘缩放）

**原因**：
如果一次性并行修改这四层，最容易出现：

- 回调接口反复返工
- 状态边界混乱
- 线程回传把当前 UI 状态覆盖掉
- 一个子系统尚未稳定时，另一个子系统建立在错误抽象上继续扩展

**代价**：
- 功能上线顺序会更分阶段
- 不能立即同时满足全部需求

**收益**：
- 每一阶段都能单独验证
- 更容易定位性能与稳定性问题
- 降低大范围回归风险

### ADR-002：当前优先阶段为 Phase A（后台加载 + 稳定文件列表模型）

**日期**：2026-03-26
**状态**：已采纳

**决策**：
优先设计与实现 Phase A：

- 后台目录加载
- 稳定的数据模型更新机制
- 避免阻塞 UI
- 为后续交互层提供可靠基础

**原因**：
交互层、收藏夹、设置、窗口行为都建立在“目录切换与列表刷新必须稳定”这个前提上。
如果 Phase A 不稳，后续任何功能都会继续放大问题。

**代价**：
- 用户可见的新交互功能不会最先出现

**收益**：
- 后续所有功能都能建立在稳定刷新机制之上
- 这是最核心的性能与稳定性基础工程

### ADR-003：快速切目录时，只接受最后一次请求的结果

**日期**：2026-03-26
**状态**：已采纳

**决策**：
当用户连续快速切换目录时，后台线程可以并发执行，但**只有最后一次请求对应的结果允许回写 UI**。旧请求结果到达后必须直接丢弃。

**原因**：
这是在重度多开文件夹、频繁切目录场景下，最稳定、最符合用户预期的策略。

它能避免：

- 旧目录结果把当前目录 UI 刷回去
- 乱序闪屏
- 选择状态和面包屑错位
- 主线程被无意义结果反复重绘打断

**不选其他方案的原因**：
- “谁先完成谁显示” 会带来严重竞态问题
- “直接做缓存优先” 适合作为后续增强，不适合作为第一层基础语义

**已知代价**：
- 会浪费部分旧任务的后台 I/O 成本
- 高频切目录时，磁盘会有额外短时压力

**后续补强方向**：
- 为后台任务设置并发上限
- 增加任务自我放弃检查
- 在稳定后再考虑目录缓存层

### ADR-004：Phase A 采用 A2「受控后台加载器」方案

**日期**：2026-03-26
**状态**：已采纳

**决策**：
Phase A 不采用“每次请求直接裸开线程再回填 UI”的最小方案，而采用 **A2：受控后台加载器**。

这意味着目录读取职责将明确拆成：

1. UI 层：只发出导航 / 刷新 / 路径跳转事件
2. BrowserState：主线程状态协调与 generation 管理
3. BackgroundLoader：后台目录读取与结果打包
4. ViewBuilder / ViewData：把原始目录结果转换成最终可绑定的 UI 列表模型

**原因**：
A2 比最小异步化方案更适合作为长期底座，因为它能更清楚地隔离：

- 谁发起目录请求
- 谁执行后台读取
- 谁负责丢弃旧结果
- 谁负责回到主线程更新模型

它尤其适合后续继续叠加：

- 排序 / 筛选
- 收藏夹
- 路径直接输入
- 目录缓存
- 更复杂的窗口行为

**代价**：
- 初始实现比最小线程化方案更复杂
- 需要重新梳理当前 `BrowserState` 的职责边界

**收益**：
- 后续扩展更稳定
- generation 与回传路径更清楚
- 更适合重度用户与高频切目录场景

### ADR-005：Phase A 的主线程状态策略为“保留旧列表 + 显式 loading + 最新结果一次性提交”

**日期**：2026-03-26
**状态**：已采纳

**决策**：
在 Phase A 中，切换目录时主线程不应立刻清空当前文件列表，而应采用以下语义：

1. 立刻更新路径与 `loading` 状态
2. 暂时保留旧列表显示
3. 等最新 generation 的结果准备好后，一次性替换列表模型
4. 若读取失败，则保留旧列表，只显示错误状态

**原因**：
这样做可以避免：

- 高频切目录时的空白闪烁
- 列表频繁清空再重建带来的卡顿感
- 失败目录被误显示成“空目录成功加载”

**代价**：
- 路径切换到新目录后，会有一个极短暂的旧内容保留窗口

**收益**：
- 主观响应更稳
- 语义更清楚：loading、成功、失败三种状态分离
- 更容易和 generation 丢弃策略配合

**补充规则**：
- 排序 / 筛选先保留在主线程视图层完成，不进入后台线程
- 同目录刷新尽量保留仍然存在的选择与焦点
- 切换到新目录时重置旧目录选择上下文

### ADR-006：Phase A 的完成标准必须同时包含逻辑、视图、异步和人工验收四层验证

**日期**：2026-03-26
**状态**：已采纳

**决策**：
Phase A 不以“能异步读目录”作为完成标准，而以四层验证共同通过作为完成标准：

1. 纯逻辑测试
2. 视图转换测试
3. 异步集成测试
4. 人工交互验收

**原因**：
仅仅把 `read_dir` 换成后台线程，并不能证明：

- 旧结果不会回刷 UI
- loading 状态不会错乱
- 排序 / 筛选 / 选择状态不会失配
- 高频切目录时不会出现闪回或假死

**代价**：
- Phase A 的交付门槛更高
- 需要补更多测试与人工验收步骤

**收益**：
- 更接近真实桌面使用稳定性要求
- 能显著降低后续交互层重构时的回归风险

### ADR-007：Windows 下文件列表中文文件名不得依赖 `overflow: elide`

**日期**：2026-03-27
**状态**：已采纳

**决策**：
在 `ui/app-window.slint` 的文件列表名称列中，中文文件名显示不得默认依赖单行 `Text` 的 `overflow: elide`。当前项目应改为由外层容器负责裁剪，文件名 `Text` 保持单行但不使用 `overflow: elide`。

**原因**：
已在 Windows 人工诊断中确认：

- 名称列宽度本身存在
- 固定中文文本可以显示
- 固定宽度的 `entry.name` 中文可以显示
- 只有真实文件名那条带 `overflow: elide` 的单行 `Text` 会把中文整段吞掉，连后缀都不显示

这说明根因不在 Rust 数据链路、`SharedString`、编码或字体整体缺失，而在 Slint/Windows 的 `wrap: no-wrap` + `overflow: elide` 中文渲染路径。

**代价**：
- 长中文文件名不能依赖 Slint 内建省略号逻辑
- 需要由容器裁剪，或后续自行设计更稳的截断策略

**后续影响**：
- 以后修改文件列表名称列时，先避开 `overflow: elide`
- 若后续需要恢复省略号行为，应先单独验证 Windows 中文路径，再决定是否启用
- 当出现“状态栏中文正常、列表中文空白”时，优先检查这里

## 当前 Phase A 设计关注点

后续进入 Phase A 设计时，应重点围绕以下问题展开：

1. 后台目录扫描任务的请求编号 / generation token 机制
2. 线程与主线程之间的数据回传边界
3. 目录切换时的加载中状态与旧数据保留策略
4. 排序、筛选、选择状态在异步刷新中的一致性
5. 大目录场景下如何避免频繁整表替换造成卡顿
6. `jwalk` 接入后与现有 `DirectoryEntry` / `BrowserViewData` 的兼容方式
