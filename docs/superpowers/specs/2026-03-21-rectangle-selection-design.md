# Files Rusted 矩形框选设计

## 背景

`Files Rusted` 当前已经具备可运行的文件管理器 MVP，拥有目录浏览、导航历史、面包屑、筛选、排序、焦点与选中分离模型、多选、范围选择、键盘快捷键、右键菜单和基础文件操作。

当前最顺的演进方向不是继续堆零散按钮，而是在现有 `Rust + Slint` 架构上，把列表选择交互继续推进到更像桌面文件管理器的层级。矩形框选正好建立在现有焦点模型、多选模型和空白区域交互之上，是下一步最合适的增强点。

## 目标

在不推翻现有技术路线和整体结构的前提下，为文件列表加入首版桌面式矩形框选：

- 保持 `Rust` 负责状态和选择语义，`Slint` 负责输入和可视反馈。
- 不破坏现有单击选中、双击打开、空白处清空选择、右键菜单、键盘选择语义。
- 支持从任意非条目本体区域开始拖框。
- 支持无修饰键拖框和 `Ctrl` 拖框。
- 支持基于当前可见列表的矩形命中。
- 支持在拖框结束后稳定更新 focus 和 anchor，继续兼容已有键盘扩选语义。

## 非目标

本轮明确不做：

- 自动滚动
- 触屏适配
- 跨视口持续拖框
- 网格/缩略图视图适配
- 文件拖拽移动
- 标签页、双栏、地址栏、传输引擎等更大功能
- 借此全面重写 `src/browser.rs`

## 交互语义

### 触发条件

- 只有当左键按下点不在条目本体上时，才允许进入拖框准备态。
- 允许从空白区域、列表间隙、非条目覆盖区开始拖框。
- 条目上的单击、双击、右键菜单逻辑保持原样。

### 拖框过程

- 左键按下后，如果指针移动超过一个最小阈值，则进入拖框态。
- 拖框态期间显示半透明选择框。
- 选择框只对当前过滤和排序后的可见条目做命中判断，不影响被过滤隐藏的项。

### 选择语义

- 无修饰键：本次拖框结果直接成为当前选择结果。
- `Ctrl`：本次拖框结果基于拖框开始前的基线选择集合进行增减/并入，尽量贴近桌面文件管理器语义。
- `Shift`：本轮不定义新的拖框语义，继续保留给已有的点击/键盘范围选择，避免和 anchor 规则打架。
- 拖框结束后如果最终没有命中任何条目，则结果为空，等价于清空选择。

### focus 与 anchor

- 拖框期间，focus 仍然独立于 selected 集合存在。
- 拖框结束后：
  - 若有命中项，focus 更新为本次稳定主命中项；
  - anchor 同步更新到该 focus，保证后续 `Shift+方向键` 继续按当前选择结果扩展。
- `Ctrl+方向键`、`Space`、`Ctrl+Space` 等现有语义保持不变。

## 架构设计

### 责任边界

**Slint 负责：**

- 接收文件列表区域的指针输入
- 区分条目本体点击与非条目区域拖框入口
- 渲染拖框 overlay
- 提供可见条目在列表坐标系中的几何信息

**Rust 负责：**

- 保存拖框会话状态
- 根据拖框矩形与条目矩形做命中计算
- 处理无修饰键与 `Ctrl` 拖框选择语义
- 生成最终的 selected paths、focus path、anchor path
- 统一把拖框结果纳入现有 selection model

### 状态扩展

在当前结构下，引入明确的拖框子状态，而不是继续让临时逻辑散落在 `BrowserState` 各处。

建议新增一个 `DragSelectionSession` 概念，包含：

- 是否正在拖框
- 起点坐标
- 当前指针坐标
- 是否为 `Ctrl` 拖框
- 拖框开始前的基线选择集合
- 当前命中的可见路径集合
- 当前拖框矩形

同时引入与可见列表布局对应的数据结构，例如 `VisibleItemLayout`，用于表达：

- 条目路径
- 在列表内容坐标系中的矩形范围

这样可以把“输入几何信息”和“选择业务语义”分开，后续即使卡片高度或布局变化，也不需要把命中逻辑写死在 UI 里。

### 对 `browser.rs` 的处理策略

本轮不直接做全面拆分，而是先把拖框相关逻辑收束成独立结构和方法组，形成未来可拆分的边界。

也就是说，这一轮追求的是：

- 让拖框逻辑拥有清晰的数据结构和入口方法
- 让 `main.rs` 继续只做 wiring
- 让 `app-window.slint` 只承担输入和显示
- 为下一轮把选择模型进一步拆到 `selection_state` 等模块铺路

## 事件流设计

### UI 新增能力

在 `ui/app-window.slint` 的文件列表区域新增三类能力：

1. 拖框 overlay
   - 在拖框会话激活时显示半透明矩形和边框
   - 使用列表内容坐标系，避免和整个窗口坐标混淆

2. 指针事件上报
   - 在列表容器上报左键 down / move / up
   - 只有落点不在条目本体上时，才转入拖框入口

3. 条目几何信息上报
   - 让每个可见条目能把其列表坐标系矩形提供给 Rust
   - Rust 依据矩形进行命中，而不是依据固定行高做近似判断

### 运行流程

1. pointer down（非条目区域）
   - UI 把起点和 modifier 交给 Rust
   - Rust 创建 drag session，记录基线选择

2. pointer move
   - UI 持续上报当前位置
   - Rust 计算当前拖框矩形
   - Rust 对当前可见条目布局做命中测试
   - Rust 更新临时选择结果与 overlay 状态

3. pointer up
   - Rust 固化最终选择结果
   - Rust 清理临时拖框会话，但保留最终的 selected/focus/anchor

### 与现有空白点击语义的兼容

当前空白区域左键点击会清空选择。为了不和拖框冲突，需要改成：

- 如果只是一次空白点击且没有形成拖框：继续清空选择。
- 如果已经形成拖框：以拖框结果为准，不再额外触发那次空白点击清空。

这条规则是本次实现的关键兼容点。

## 测试策略

矩形框选的风险主要在选择语义而不是绘制本身，因此测试重点应落在 Rust 状态层。

### 单元测试

优先把选择计算整理成可测试逻辑，并覆盖以下行为：

- 无修饰键拖框命中单项
- 无修饰键拖框命中多项
- 无命中时清空选择
- `Ctrl` 拖框与原选择集合合并/切换
- 拖框结束后 focus 与 anchor 更新正确
- 被过滤隐藏的项不参与当前拖框命中

### 构建与运行验证

实现完成后至少执行：

1. `cargo build`
2. `xvfb-run -a ./target/debug/files-rusted`

如果 headless 环境下无法完整完成鼠标交互验收，应明确区分：

- 编译是否通过
- 程序是否能正常启动
- 哪些拖框交互仍需真实桌面环境补充手工验收

## 风险与控制

### 风险 1：破坏现有单击/双击/空白点击语义

控制方式：

- 只让非条目区域进入拖框入口
- 引入“是否形成真正拖框”的阈值判断
- 空白点击与拖框完成走两条明确分支

### 风险 2：几何计算耦合到 UI 细节

控制方式：

- UI 只上报条目矩形
- Rust 只消费标准化布局信息，不假设固定卡片高度

### 风险 3：继续放大 `browser.rs`

控制方式：

- 本轮新增结构和方法按“选择子系统”收口
- 为后续拆分 selection state 提供天然边界
- 不借机改动无关导航、文件操作逻辑

## 实施结论

本轮应实现的是：

- 一版桌面式矩形框选
- 严守 `Rust 管状态 / Slint 管输入与显示` 的边界
- 把拖框逻辑整理为未来可拆分的选择子结构
- 不把这次任务扩展成大规模重构或新功能开发

这能在最小破坏现有 MVP 的前提下，把 `Files Rusted` 的列表交互向更接近原版 Files 的方向推进一步。
