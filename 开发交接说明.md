# Files Rusted 开发交接说明

## 1. 文档目的

本文档用于把当前讨论结果沉淀成一份可交接的中文说明，供后续开发人员快速接手 `Files Rusted` 项目。

这份文档重点说明：

- 为什么新建 `Files Rusted`
- 为什么选择 `Rust + Slint`
- 当前已经完成了什么
- 当前环境限制是什么
- 后续应该先做什么

## 2. 项目背景

我们先对开源项目 `Files` 做了一轮静态代码审查，并形成了项目级判断：

- 原项目规模较大
- 工程复杂度高
- WinUI 3 / XAML 前端交互复杂
- 文件列表、分组、排序、状态刷新链路较重
- 后续如果继续基于原工程做深度开发或重构，门槛和维护成本都偏高

原始审查结论已经整理在：

- [Files-优化计划.md](/app/work8/Files-优化计划.md)

基于这轮判断，我们没有继续沿用原有 `WinUI 3 + XAML + C#` 技术路线来做新的实验项目，而是决定新开一个目录，单独推进新实现。

## 3. 当前决策结论

### 3.1 新项目名称

新项目目录名称：

- [Files Rusted](/app/Files%20Rusted)

### 3.2 技术栈选择

当前明确的技术路线是：

- 后端：Rust
- 前端：Slint

### 3.3 选择原因

当前讨论形成的共识如下：

- 纯 GPU 渲染路线太新，工程风险偏高。
- 我们希望前端表现稳定、性能更容易控制。
- `Slint` 比较适合作为当前阶段的桌面 UI 方案。
- `Rust` 更适合承担文件系统、并发任务、状态管理、性能敏感逻辑。
- 这条路线整体难度比“直接采用更激进的新图形栈”更低，更容易进入可持续开发。

简化结论：

- 我们不是在追求最激进的图形方案。
- 我们是在追求“性能、稳定性、开发复杂度”三者之间更平衡的路线。

## 6. 当前环境限制

这次工作是在当前容器环境中完成的，存在如下限制：

- 当前环境最初没有安装 `cargo` / `rustc`，本轮已经补装
- 当前使用 `rustup` stable 工具链
- 当前没有真实桌面显示环境，因此运行验证依赖 `xvfb-run`
- 目前已经完成 `cargo test`、`cargo build` 和启动级验证，但还没有做完整桌面交互测试

结论：

- 当前项目已不只是骨架，而是“已编译验证、已做基础启动验证的文件管理器 MVP”

## 8. 建议的开发方向

### 8.1 第一阶段目标

建议不要一开始就追求完整复刻 `Files`，而是先把当前 MVP 打磨稳定。

第一阶段推荐目标：

- 保持应用壳层稳定
- 完善当前浏览与文件操作体验
- 明确 Rust 状态层与 Slint UI 的边界
- 建立可扩展的数据模型

### 8.2 第一阶段建议拆分

推荐按以下顺序推进：

1. 导航与文件操作的剩余状态边界整理
2. 真实桌面环境下的交互验收与细节打磨
3. 更细的选择语义增强
4. 再逐步补文件操作能力

更具体地说：

1. 继续收敛核心数据结构
   - 文件项
   - 目录状态
   - 当前路径
   - 排序参数
   - 筛选参数

2. 补齐当前文件管理器 MVP 的剩余交互与稳定性细节
   - 打开目录
   - 列表展示
   - 进入目录
   - 返回上级目录
   - 刷新当前目录
   - Back / Forward
   - 面包屑导航
   - 排序
   - 筛选

3. 再实现交互增强
   - 选中
   - 多选
   - 导航历史
   - 面包屑
   - 右键菜单
   - 双击进入

4. 最后再逐步加入写操作
   - 更完整的复制 / 移动
   - 更完整的新建能力
   - 更完整的文件打开与系统集成
   - 复制 / 移动

## 9. 当前阶段不建议立即做的事

为了避免一开始就把项目做重，当前不建议优先投入这些内容：

- 完整复刻原版 `Files` 的所有能力
- 过早引入复杂多线程刷新模型
- 一开始就接入 Git、FTP、压缩包、预览系统、回收站等复杂子系统
- 过早优化成高度组件化但尚未稳定的架构

当前更合理的路径是：

- 先做稳定可运行的浏览器壳层
- 再逐步增加能力

## 10. 架构建议

后续接手开发时，建议尽早明确这两个边界：

### 10.1 Rust 负责什么

建议 Rust 侧负责：

- 文件系统访问
- 数据模型
- 状态管理
- 异步任务
- 缓存与后续性能优化

### 10.2 Slint 负责什么

建议 Slint 侧负责：

- 视图布局
- 交互绑定
- 列表展示
- 基础界面反馈

### 10.3 关键原则

建议避免把太多业务逻辑堆回 UI 层。

这也是我们从原 `Files` 项目审查中得到的一个重要经验：

- 真正容易失控的，不是 UI 框架本身
- 而是状态同步链路太重、职责边界不清、热路径逻辑分散

## 11. 当前建议优先级

### P0

- 在真实桌面环境中验证主窗口可正常交互
- 确认 `Slint` 版本与工具链长期兼容
- 补做一次手工浏览验证，确认目录切换和列表展示符合预期

### P1

- 继续拆分当前 `browser.rs` 中与导航、目录刷新、文件操作相关的剩余职责
- 在真实桌面环境中补做一次拖框自动滚动手工验证
- 建立更明确的路径状态和文件项数据结构

### P2

- 继续细化导航历史、面包屑和选中模型
- 继续细化多选和快捷键行为
- 加强焦点态和选中态的视觉区分
- 打磨矩形框选和拖框自动滚动的边界体验
- 把当前基础右键菜单扩成真正可用的操作菜单
- 让 UI 和状态层的边界稳定下来

### P3

- 再逐步补更完整的文件操作
- 接入系统剪贴板或更明确的传输状态展示
- 细化文件打开失败时的用户反馈
- 再评估更复杂能力，例如预览、标签、Git、压缩包等
